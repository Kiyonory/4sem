# Лаба 5 (Django 4, часть 2) — что сделано и ответы на вопросы

Демонстрация — **только в админке Django** (отдельный HTML-фронтенд не используется).

## Демонстрации в проекте

| Требование | Где |
|------------|-----|
| **ImageField** | `Tag.icon`, `Product.cover_image` — `shop/models.py`; загрузка и просмотр в админке (теги и товары). |
| **FileField** | `Product.specification_file` — загрузка PDF и др. на странице товара в админке; ссылка на файл в форме редактирования. |
| **redirect** | После сохранения объекта в админке Django отдаёт **редирект** (302) на список или форму — стандартное поведение `ModelAdmin`. Сообщения об успехе — через `message_user` в действиях (например, подсчёт выбранных товаров). |
| **PDF в админке** | Действие **«Скачать PDF по выбранным товарам»** — `generate_pdf_catalog` в `shop/admin.py` (ReportLab: `SimpleDocTemplate`, `Paragraph`). |
| **Действия администрирования** | 1) PDF по выбранным товарам. 2) **«Показать число выбранных товаров»** — `show_selected_products_count` (сообщение пользователю). |
| **Метод модели** | `Product.title_with_brand()` — строка «Бренд — название»; используется в PDF, в списке админки (`title_brand_short`). |
| **requirements.txt** | Добавлены `Pillow` (для ImageField) и `reportlab` (для PDF). |

### Кириллица в PDF (квадраты на Mac / iPhone)

Встроенный шрифт ReportLab (Helvetica) **не содержит кириллицу** — просмотрщики показывают квадраты. В проекте подключается **TTF с кириллицей** (`shop/pdf_fonts.py`): по очереди проверяются `shop/fonts/DejaVuSans.ttf`, системные Arial / Arial Unicode (macOS, Windows), DejaVu (Linux). Положите **DejaVuSans.ttf** в `shop/fonts/` (инструкция в `shop/fonts/README.txt`). После этого PDF с русским текстом отображается везде нормально.

### Настройки медиа

- `settings.py`: `MEDIA_URL`, `MEDIA_ROOT`.
- `config/urls.py`: при `DEBUG=True` раздача файлов через `static(MEDIA_URL, document_root=MEDIA_ROOT)`.
- Загруженные файлы попадают в каталог `media/` (лучше не коммитить в git — см. `.gitignore`).

## Вопрос: собственный функциональный метод в модели (учебник ~стр. 410)

**Ответ:** Это обычный метод экземпляра модели (не менеджер и не `Meta`), который инкапсулирует логику и возвращает значение для отображения или расчётов.

**В проекте:** `Product.title_with_brand()` в `shop/models.py` — возвращает строку `«Бренд — название»`. Его вызывают админка (`obj.title_with_brand()` в `title_brand_short`), действие генерации PDF.

Такой метод не хранится в БД, а вычисляется при обращении; при необходимости внутри можно обращаться к полям и связям (`self.brand`, `self.name`).

## Вопрос: File Uploads — особенности сохранения файлов в формах в вашей архитектуре

**Ответ (кратко для защиты):**

1. **В модели** поля `FileField` / `ImageField` хранят в БД **относительный путь** к файлу; сами файлы лежат в `MEDIA_ROOT` под `upload_to=...`.

2. **В форме** для отправки файлов обязательно **`enctype="multipart/form-data"`**. В **админке Django** для моделей с `FileField`/`ImageField` это подключается автоматически.

3. **Во view** при обработке POST с собственной формой нужно передать **`request.FILES`** вместе с `request.POST`. В админке это делает `ModelAdmin`.

4. **Сохранение:** `form.save()` записывает файл на диск и путь в БД.

5. **Продакшен:** раздача пользователям через веб-сервер (nginx) или `django-storages` для облака; в разработке — `static()` для `MEDIA_URL` при `DEBUG=True`.

6. **Безопасность:** проверять тип и размер файлов, не доверять имени файла из запроса целиком.
